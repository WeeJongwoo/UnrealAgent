② 툴 등록 흐름 — 두 경로, 하나의 목적지

핵심 원리

Backend의 ToolRegistry.Tools 딕셔너리에는 두 종류의 도구가 동일한 모양으로 들어간다: (name, IAgentTool instance, InputSchema). 호출자 입장에서는 구분 없이 이름만 보고 실행하며, 실제 구현은 다형성으로 분기된다.

🧩 경로 A — AgentTool (C# 내장)

[AgentTool("web_fetch", "Fetch a URL")]
public sealed class WebFetch : AgentTool<WebFetch.Input>
{
    public sealed record Input(
        [property: JsonPropertyName("url")]
        [property: Description("Target URL")]
        string Url);

    protected override Task<ToolResult> ExecuteAsync(
        Input In, AgentSession S, CancellationToken Ct) { ... }
}

스키마 생성은 record 프로퍼티의 [JsonPropertyName] + [Description] + nullable 판정으로 자동 수행.

🧩 경로 B — MCP Tool (UE 원격)

USTRUCT(meta=(McpTool="execute_python"))
struct FExecutePythonTool : public FMcpTool
{
    UPROPERTY(meta=(ToolParam="code", Required,
                    Description="The Python code to execute"))
    FString Code;

    UPROPERTY(meta=(ToolParam="purpose", Required,
                    Description="짧은 한국어 설명"))
    FString Purpose;

    virtual FString ToolDescription() const override { ... }
    virtual FMcpResponse Execute() override;
};

UE 자동 수집은 리플렉션이 전부. 새 도구 추가 시 어떤 레지스트리 코드도 고칠 필요가 없음.

공통 목적지: Tools 딕셔너리

실행 시점에는 Tools[name].Tool.ExecuteAsync(...) 한 줄로 호출. 다형성 덕분에 AgentTool/McpProxyTool 분기를 신경쓰지 않음.

③ 사용자 메시지 처리 흐름

사용자가 채팅창에 입력 → Claude가 도구 호출 → UE에서 Python 실행 → 결과 반환 → 최종 답변까지. 각 단계에서 실제 어떤 JSON이 흐르는지 전부 수록.

전제: 사용되는 3가지 메시지 프로토콜

🟢 MCP = JSON-RPC 2.0 (Backend ↔ UE)

모든 요청/응답이 공통 봉투(envelope)를 씀. 메서드는 3개만 사용: initialize, tools/list, tools/call.

// 요청 봉투
{
  "jsonrpc": "2.0",    // 프로토콜 버전 (항상 "2.0")
  "id": <int>,          // 요청-응답 매칭용 ID
  "method": <string>,   // 호출할 메서드 이름
  "params": { ... }     // 메서드 파라미터 (객체/배열)
}

// 성공 응답
{
  "jsonrpc": "2.0",
  "id": <int>,          // 요청 id와 동일
  "result": { ... }     // 메서드별 결과
}

// 에러 응답
{
  "jsonrpc": "2.0",
  "id": <int>,
  "error": {
    "code": -32601,     // 표준 에러 코드
    "message": "Method not found"
  }
}

표준 에러 코드: -32700 Parse error · -32600 Invalid Request · -32601 Method not found · -32602 Invalid params · -32603 Internal error

🟠 Claude Messages API (Backend ↔ Claude)

HTTPS + SSE 스트리밍. content는 여러 블록의 배열 — text, thinking, tool_use, tool_result.

// 요청
POST /v1/messages
{
  "model": "claude-opus-4-5",
  "max_tokens": 8192,
  "system": "...",
  "tools": [ ... ],     // input_schema 포함
  "messages": [ ... ],  // role + content 블록 배열
  "stream": true
}

// 스트림 이벤트 (SSE)
event: message_start
event: content_block_start   // block index + type
event: content_block_delta   // 부분 텍스트/도구 인자
event: content_block_stop
event: message_delta         // stop_reason 등
event: message_stop

stop_reason: end_turn(정상 종료) · tool_use(도구 호출 필요) · max_tokens · pause_turn(잘림, 이어쓰기)

🟣 Backend 내부 이벤트 (AgentLoop ↔ UI)

Claude 스트림 이벤트를 ApiStreamSpan이 파싱하여 UI에 최적화된 ChatEvent로 재방출.

abstract record ChatEvent
{
    record User(string Text);               // 사용자 입력 에코
    record Assistant(string TextDelta);     // 텍스트 델타 (실시간 스트리밍)
    record Thinking(string Delta);          // 확장 사고 델타
    record ToolStart(string Id, string Name, string InputJson);
    record ToolEnd(string Id, ToolResult Result);
    record System(string Message);          // 시스템 알림
    record Done();                          // 턴 종료
}

    public static ToolResult Success(string Text) => new() { IsError=false, Text=Text };
    public static ToolResult Error(string Text)   => new() { IsError=true,  Text=Text };
}

핵심 개념 요약

  • MCP (Model Context Protocol) — Anthropic 표준. JSON-RPC 2.0 위에서 initialize / tools/list / tools/call 3개 메서드로 도구를 노출.
  • 리플렉션 자동 등록 — UE는 TObjectIterator<UScriptStruct>로, C#은 Assembly.GetTypes() + GetCustomAttribute로. 새 도구 추가 시 레지스트리 코드 수정 없음.
  • 프록시 패턴 — McpProxyTool이 IAgentTool을 구현하여 MCP 원격 호출을 로컬 도구처럼 취급. 호출자는 차이를 모름.
  • while(true) 에이전트 루프 — Claude가 tool_use를 보낼 때마다 도구 실행 후 messages에 tool_result를 추가하고 재호출. end_turn까지 반복.
  • 트랜잭션 안전성 — Python 실행은 GEditor->BeginTransaction으로 감싸 실패 시 에디터 상태 롤백 + Undo 지원.
